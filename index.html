Deep Agents — Your Research Team

Deep Agents are AI systems that can plan, research, delegate, and remember. Give them a complex question and they handle the rest — breaking it into steps, searching the internet, and writing polished findings to files.

The 4 Pillars — What Makes an Agent "Deep"

A regular chatbot is "shallow" — it answers one question and forgets you. A deep agent has four superpowers:

1

Planning

Breaks big tasks into a checklist of steps, then works through them one by one. If step 3 fails, it adjusts the plan.

2

Sub-Agents

Creates helper agents — like assigning a junior analyst to research one company while it researches another. They work in parallel.

3

File System

Reads and writes files. Long research gets saved to a document. Data gets organized. Nothing is lost when the conversation gets long.

4

Memory

Remembers things between conversations. Your preferences, past research, project context — it carries forward.

Gas Town — Your AI Company

Created by Steve Yegge (legendary engineer, ex-Google/Amazon). Gas Town is "Kubernetes for AI coding agents" — it manages a fleet of Claude Code instances. Named after Mad Max: Gas Town keeps everything fueled and running.

What Gas Town Actually Is

Imagine you could hire 20–30 software engineers who never sleep, never argue, and can start working the moment you describe what you want built. That's Gas Town. One of them — the Mayor — acts as your project manager. You describe what you want, the Mayor breaks it into tasks, assigns them to workers (called Polecats), and the workers all code in parallel on separate branches. When they're done, a merge system combines everything and checks for conflicts. You just watch it happen.

The Seven Roles Explained

Mayor Your main contact. Coordinates everything. You talk to the Mayor, and the Mayor talks to everyone else.
Polecats Worker agents. Each picks up a task, writes code, and submits a merge request. There can be many running at once.
Crew Persistent agents for collaborative design work. "gib" is you — a crew member who can guide the work.
Refinery The merge queue. Combines everyone's code and checks for conflicts before anything lands on the main branch.
Witness Per-rig health monitor. Watches workers and flags stuck or crashed ones so the Mayor can reassign their work.
Deacon Town-level watchdog. Runs regular health checks and maintenance routines across the entire workspace.
Dogs Maintenance crew. Cleanup, garbage collection, and housekeeping to keep things running smoothly.

Key Concepts in Plain English

Town Your workspace folder (~/gt/). Everything lives here — projects, agents, work items.
Rigs Projects. Each rig wraps a git repo with its own agents. You have 1 rig: "portfolio".
Beads Work items — like Jira tickets stored in git. Each bead is a task an agent can pick up. Format: po-abc12
Convoys Groups of beads bundled together. "Build this feature" = a convoy of 5 beads.
Hooks Persistent storage for each agent. If an agent crashes, its hook has all its work so it can resume.
GUPP "Gas Town Universal Propulsion Principle" — if there's work on your hook, YOU MUST RUN IT. This keeps agents from getting stuck.
Sling The command to assign work to an agent. You "sling" a bead to a polecat.
Seance Talk to previous agent sessions to recover context. Like replaying a past conversation.

How It Actually Works — A Deep Dive

What really happens behind the scenes when you say "Build me a dashboard" to the Mayor. Every step, from English sentence to merged code.

The Full Lifecycle: From Idea to Merged Code

1

You Describe What You Want

You attach to the Mayor with gt mayor attach and type something like "Build me a dashboard with an overview page, a settings page, and a data table." The Mayor is a Claude Code instance with special Gas Town context — it knows all the roles, all the commands, and your project's codebase.

2

The Mayor Creates a Plan

The Mayor analyzes your request and breaks it into discrete, parallelizable tasks. It creates beads (work items) for each one — "Build the overview page", "Build the settings page", "Build the data table component." These beads are grouped into a convoy (a batch of related work). The plan is stored in Dolt, Gas Town's built-in SQL database, so nothing is lost if a session crashes.

3

Polecats Are Spawned

For each bead, the Mayor spins up a polecat — a fresh Claude Code instance running in its own tmux window with its own git worktree. This is key: each polecat works on a separate copy of your codebase, so they never step on each other's files. The polecat's hook (persistent state) records exactly which bead it is working on, so if it crashes, it knows where to resume.

4

Parallel Coding Begins

All polecats start coding simultaneously. Polecat 1 builds the overview page while Polecat 2 builds settings and Polecat 3 builds the data table. They read the codebase, write files, run tests, and iterate — just like a human developer would, but each in their own isolated branch.

5

The Witness Monitors Everything

While the polecats work, the Witness (a dedicated monitoring agent for each rig) runs periodic health checks. It looks for stuck agents (no commits in too long), crashed sessions (tmux window gone), or agents that seem to be going in circles. If something is wrong, the Witness notifies the Mayor, who can reassign the work to a new polecat.

6

Polecats Submit Merge Requests

When a polecat finishes its bead, it submits a merge request — pushing its branch and marking the bead as ready for review. The polecat's work is done; it can be assigned new work or shut down.

7

The Refinery Merges and Verifies

The Refinery is Gas Town's automated merge queue. It picks up merge requests one at a time, checks for conflicts against the main branch, runs any tests, and attempts the merge. If there's a conflict, the Refinery can either resolve it automatically or create a new bead for a polecat to fix it manually. Only clean, passing code lands on your main branch.

8

You Get the Result

The Mayor notifies you that the convoy is complete. All three pages have been built, reviewed, and merged into your main branch. You pull the latest code and see the finished dashboard. The entire process — from "build me a dashboard" to merged code — might have taken 15 minutes instead of the 3 hours it would take one developer.

What Happens When Things Go Wrong

Agent crashes The agent's hook has all its state. When the Witness detects the crash, it tells the Mayor. The Mayor spawns a fresh polecat that reads the hook and picks up exactly where the previous one left off. No work is lost.
Agent gets stuck The Witness notices no progress (no new commits, no status updates). It sends a nudge. If the agent still doesn't respond, the Mayor issues a death warrant (via gt polecat nuke), kills the stuck agent, and reassigns the bead.
Merge conflict The Refinery detects the conflict and either auto-resolves it (for simple cases) or creates a new bead specifically for conflict resolution. A polecat picks it up, resolves the conflict in its own worktree, and resubmits.
Tests fail The Refinery rejects the merge request and creates a bead describing the failure. A polecat is assigned to fix the failing tests before the code can land.
Your Mac sleeps tmux sessions survive sleep. When your Mac wakes up, all agents resume. The GUPP principle ("if there's work on your hook, you MUST run it") means agents automatically pick up where they left off.

Frequently Asked Questions

Quick answers to the most common questions about Gas Town.

Is this expensive?

Each polecat is a Claude Code session, so costs scale with the number of active workers. A single polecat might cost a few dollars per complex task. Running 10 polecats in parallel for an hour could cost $20-50 depending on the work. Use gt costs to monitor spending in real time, and set budget limits with the Mayor. For many teams, the time savings far outweigh the API costs — what takes one developer a day, Gas Town can do in 30 minutes.

Can I use this with my existing projects?

Yes. Any git repository can become a Gas Town rig with gt rig add. Gas Town does not modify your repo structure — it creates worktrees alongside it and manages branches. Your existing code, CI/CD, and deployment stay exactly the same. Gas Town just adds a layer of orchestration on top.

What if an agent gets stuck?

The Witness detects stuck agents automatically (no commits or updates for too long). It nudges the agent first. If that doesn't help, the Mayor reassigns the bead to a fresh polecat. As a last resort, you can run gt polecat nuke to force-kill a stuck agent. The hook system means all progress is saved — the replacement polecat picks up where the stuck one left off.

Do I need to know how to code?

For initial setup (installing Homebrew, running terminal commands), some comfort with the terminal is needed. But once Gas Town is running, your day-to-day interaction is just talking to the Mayor in plain English. "Build me a login page", "Fix the spacing on the homepage", "Add dark mode to the settings page." The Mayor handles all the technical details — creating branches, writing code, merging, and testing.

How is this different from GitHub Copilot?

Copilot autocompletes code as you type — it is a writing assistant. Gas Town is a team of autonomous agents that can build entire features without you touching the keyboard. Copilot helps you write one file faster. Gas Town writes 20 files simultaneously across multiple agents, merges them, and runs tests. They serve very different purposes and can be used together.

Can I review the code before it merges?

Absolutely. You can configure the Refinery to require human approval before merging, or you can review merge requests manually. Many people start with auto-merge to see how it works, then add approval gates for production-critical code. The Mayor can also be told "always ask me before merging" in your initial instructions.

What happens if my computer shuts down?

tmux sessions survive sleep, so closing the lid is fine. For a full shutdown or restart, run gt down to gracefully stop everything. When you boot back up, gt up restarts all services. Every agent's state is persisted in its hook and in the Dolt database, so no work is lost. The GUPP principle ensures agents automatically resume their assigned work.

How many projects can Gas Town manage at once?

There is no hard limit on rigs. Each rig gets its own Witness, its own worktrees, and its own pool of polecats. The practical limit is your machine's resources (CPU, RAM, disk) and your Claude API budget. Most users run 1-3 rigs comfortably on a modern Mac.
